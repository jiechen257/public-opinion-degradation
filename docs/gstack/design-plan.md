## Problem Statement

`agent-hub` 不是要再做一个“agent 版知乎”或“AI 版微博”。

这个项目要解决的问题是：普通网友虽然每天都在经历舆论劣化，但大多数时候只能在事后模糊地感觉到“环境越来越差”，却很难直观看见一个讨论是如何一步步从正常交流滑向情绪化、极端化、站队化，最终吞掉平台“帮助交流、提供信息”的初衷。

因此，这个原型的目标不是承载真实社区，而是把这种劣化过程压缩、显影并可视化。用户进入产品后，应当像看一场可交互的舆论演化实验：同一个普通问题，在不同 agent 性格组合之下，经过 10 轮推进，最终演变成 3 个阶段，并留下若干句极具现实熟悉感的“经典言论”。

## What Makes This Cool

这个产品最有冲击力的点，不是“有很多 AI 在说话”，而是下面四件事同时成立：

1. 它看起来像大家熟悉的知乎/微博式讨论区，理解门槛极低。
2. 它让不同 agent 真正像不同类型的人，而不是几段换皮回答。
3. 它不是静态展示观点，而是让用户看见讨论如何被人格差异、情绪奖励和话术模仿一步步带偏。
4. 它最后产出的“经典言论”会让用户产生强烈的既视感：`这话我见过，我甚至说过。`

这让它更像一个“公众理解装置”而不是一个内容平台，也让它有天然的传播力和展示力。

## Constraints

- 当前阶段只做本地 mock 和产品原型，不承担真实社区运营。
- 目标受众是普通网友，不是研究者、治理人员或专业分析师。
- 首版要先让用户看懂“过程”，再追求更复杂的玩法。
- 讨论主题必须足够普通、足够日常，才能让用户快速代入。
- 产品需要参考一篇关于舆论演变历史的文章，但当前仓库内 [article.md](/Users/jiechen/per-pro/agent-hub/article.md) 还是空文件，因此首版只能先按创作意图建模，后续再把文章中的历史阶段、典型角色和典型句式映射进来。
- 产品需要尽可能完整，不应停留在“几条 agent 回复”的浅层 demo，而应明确轮次、阶段、归因和可复玩结构。

## Premises

1. 首版不是新社区，而是一个可观看、可切换、可复跑的舆论演化体验。
2. 核心体验必须是同一个问题经过 `10` 轮推进，最后归纳为 `3` 个阶段。
3. 用户价值不在于“答案是否聪明”，而在于这些表达是否真实像人、是否能激起现实既视感。
4. 人格模板不是点缀，而是演化引擎的输入条件；不同人格组合应明显改变演化轨迹。
5. 平台外观应借用讨论社区的熟悉感，但交互目的不是让用户继续参与争论，而是观察机制如何发生。

## 外部认知校准

### Layer 1：大家已经知道什么

- 讨论平台会被情绪、站队和奖励机制带向极化，这已经是大众经验层面的共识。
- 点赞、传播、围观等反馈机制往往会鼓励更激烈、更具有表演性的表达。

### Layer 2：外部研究和当前方向在说什么

- Yale 2021 指出，社交平台上的奖励机制会训练用户使用更愤怒的表达。
- Nature 2024 指出，讨论越长、互动越密，毒性更容易上升，而且这种模式跨平台存在。
- 多 agent 社会模拟研究已经开始用人格差异、社会规则和多轮互动来模拟舆论传播，但多数服务于研究、治理和实验。

### Layer 3：本项目的第一性原理判断

主流方向是在问“怎么治理坏舆论”或“怎么建设更理性的讨论系统”；本项目真正特别的地方，是把它变成一个让普通人直接感知机制的体验产品。

**EUREKA：** 这里不该优先做“更好的平台”，而应该先做“让人看见平台为什么会坏掉”的显影器。只有当普通网友能看见机制，后面的反思、传播、讨论才有抓手。

## Approaches Considered

### Approach A: 定轨演化剧场

把若干个精选话题做成固定 `10` 轮脚本，用户主要观看、切换阶段、查看总结。

- 优点：最容易把节奏和感染力打磨到位，最容易迅速展示。
- 缺点：复玩性和组合空间不足，不足以承载“人格模板”这一核心野心。

### Approach B: 人格模板 + 演化引擎

用户选择一个普通问题，再选择若干 agent 人格模板；系统根据人格、轮次状态和阶段规则生成 `10` 轮互动，并在结尾输出 `3` 个阶段总结与“经典言论”回看。

- 优点：完整承接项目野心，既能演示，又能复玩，还能不断扩展模板、话题和阶段规则。
- 缺点：需要更明确的状态设计和生成控制，不然会退化成散乱聊天。

### Approach C: 经典言论博物馆

把重点放在不同人格和不同阶段会出现的典型话术上，让用户像逛展一样探索和对照。

- 优点：传播性强，截图效果强，教育性直接。
- 缺点：连续过程感较弱，不够像“讨论一步步坏掉”。

## Recommended Approach

选择 **Approach B: 人格模板 + 演化引擎**。

原因不是它“功能最多”，而是它最完整地保留了项目的核心张力：

- 有熟悉的平台外壳，用户秒懂。
- 有人格模板，讨论才像真人。
- 有多轮推进，才看得见滑坡过程。
- 有阶段归纳，用户才记得住。
- 有经典言论回看，感染力才会留下来。

推荐的首版体验结构如下：

1. 用户进入首页，先看到一个普通、熟悉的问题样本。
2. 用户选择或接受默认的 3-4 个 agent 人格模板。
3. 系统播放或逐轮展开 `10` 轮演化。
4. 用户在过程中能看见当前轮次属于哪个阶段、哪种人格在放大局势、哪些表达被平台反馈奖励。
5. 结束后系统输出：
   - `3` 个阶段总结
   - 本次演化的关键转折点
   - 若干句“经典言论”
   - 重新换人格组合再跑一次的入口

这个结构既满足普通网友第一次使用时的观看体验，也给后续的“换人格、换话题、换阶段规则”留足空间。

## 系统结构建议

### 1. 话题种子层

负责提供“足够普通、足够容易代入”的问题样本。

要求：
- 主题不是宏大抽象议题，而是普通网友日常能遇到的现实问题。
- 每个问题应天然带有价值冲突、现实压力或身份张力，便于触发不同人格的典型反应。

示例：
- 要不要为了稳定接受不喜欢的工作
- 该不该为了孩子牺牲个人发展
- 年轻人抱怨工作压力是不是矫情

### 2. 人格模板层

负责定义 agent 的表达风格、价值取向、攻击方式、信息处理倾向和情绪阈值。

首版建议每个模板包含：

- `name`：人格名
- `core_belief`：核心信念
- `tone`：语气风格
- `trigger_points`：最容易被什么刺激
- `argument_pattern`：常用论证套路
- `escalation_pattern`：进入下一阶段后如何变形
- `classic_lines_seed`：常见话术种子

首版最少应提供 6-8 个高辨识度模板，例如：
- 道德裁判官
- 情绪放大器
- 冷眼犬儒者
- 经验分享者
- 事实纠正者
- 群体代言人
- 热点搬运工
- 讽刺型旁观者

### 3. 演化引擎层

负责让对话不是随机聊天，而是有明确滑坡结构。

建议把 `10` 轮分成 `3` 段：

- `Round 1-3`：正常讨论
  - 仍在回应问题本身
  - 允许经验、事实、建议并存
- `Round 4-7`：立场固化
  - 开始标签化、身份化、道德化
  - 对话目标从理解问题转向占据道德高地或争夺叙事
- `Round 8-10`：舆论失控
  - 事实退场，情绪和表演接管
  - 原问题沦为引战容器

引擎的输入：
- 话题种子
- 人格组合
- 当前轮次
- 当前阶段
- 前序对话摘要
- 平台反馈信号（哪些表达被强化）

引擎的输出：
- 当前轮次若干条发言
- 本轮关键推动人格
- 本轮偏移方向
- 是否进入下一阶段

### 4. 平台反馈层

这是项目非常重要的一层，不能省略。

如果只有人格，没有“平台奖励什么”的信号，产品就会更像多 agent 戏剧，而不是平台舆论演化。

首版至少需要模拟：
- 高情绪表达更容易被看见
- 羞辱性、绝对化表达更容易被跟进
- 温和、具体、复杂的表达容易被淹没

这层不必做成复杂的算法系统，但必须在界面和总结中被用户感知到。

### 5. 经典言论提取层

每轮对话结束后，不只是保存“谁说了什么”，还要提取出那些最具现实既视感的话术。

这些内容未来可以独立变成：
- 传播截图
- 经典话术合集
- 阶段对照卡片
- 不同人格模板的代表表达

这是产品后续传播力和记忆点的关键资产。

### 6. 阶段归因层

结束时不能只告诉用户“现在很糟”，而要解释：

- 是哪一轮开始转向的
- 是哪类人格推动了转向
- 平台哪种奖励机制加速了这一过程
- 哪些原本正常的表达被逐渐排挤掉

这层是“看热闹”与“看懂机制”的分界线。

## 用户路径

### 首次使用路径

1. 进入首页，看见一个非常普通的问题。
2. 选择默认或自定义人格组合。
3. 点击开始观察。
4. 跟随 10 轮演化逐轮展开。
5. 在每个阶段看见当前讨论氛围如何变化。
6. 结束后查看 3 阶段总结和经典言论。
7. 被鼓励重新换一组人格重跑同一问题。

### 复玩路径

1. 保持同一问题不变，只替换人格组合。
2. 对照不同组合的结果。
3. 观察哪些人格组合最容易把讨论推入极端。

### 后续增强路径

1. 用户自定义人格。
2. 用户保存和分享某次演化截图。
3. 用户对比不同问题在同一人格组合下的演化差异。

## 界面与交互原则

- 首页第一屏必须先演示、后控制：
  第一视图优先展示一个精选问题、双世界即将分叉的预告和唯一主 CTA，不在首屏堆叠完整参数控制区。
- 默认像讨论平台，但目标是“观察”而不是“加入战场”。
- 中央区域必须始终是时间线主舞台。
- `3` 个阶段要始终可见，帮助用户压缩认知负担。
- “当前是怎么坏掉的”要比“当前谁说得更聪明”更醒目。
- 每轮内容不宜过长，重点是连续推进和对比感。
- 结束页必须有一个强总结，不然体验会像普通聊天记录。

## 设计补充：信息架构

### 首页信息层级

1. 第一眼：一个足够普通、足够刺痛的精选问题，以及“双世界会如何分叉”的预告。
2. 第二眼：这是一个舆论显影实验，而不是普通论坛或 AI 聊天页。
3. 第三眼：开始实验的主 CTA。
4. 第四眼：次级控制项，例如话题模板改写、人格组合和热度奖励开关。

首页首屏不应先暴露完整控制台。控制项属于次级信息，放在首屏以下或折叠区中，防止产品被误读为“参数玩具”。

### 结果页信息层级

1. 第一眼：双世界差异摘要，说明两个世界最终为什么走向不同结果。
2. 第二眼：关键拐点与“第一句带偏的话”，帮助用户快速理解因果转折。
3. 第三眼：反事实救援结论，说明如果改掉某个条件，局势本来会如何变化。
4. 第四眼：完整的 10 轮时间线，作为证据层展开。

结果页不是聊天记录页，而是“先给结论，再给证据”的调查报告结构。完整时间线应服务于解释，而不是抢占第一视觉层级。
结果页顶部三大模块的固定顺序为：
`双世界差异摘要 -> 关键拐点 -> 反事实救援`。

## 设计补充：用户旅程与情绪弧线

| Step | User Does | User Feels | Plan Specifies? |
|------|-----------|------------|-----------------|
| 1 | 进入首页，看到一个普通但刺痛的问题 | “这说的就是我见过的争论” | 首页第一屏只展示精选问题、双世界预告和主 CTA，先建立代入感 |
| 2 | 看到这不是普通论坛，而是一场舆论显影实验 | “原来我不是来刷内容，而是来看机制怎么发生” | 首页第二层明确说明产品是显影器而不是聊天页 |
| 3 | 点击开始实验，进入叙事式等待 | “一场实验正在成形，而不是页面卡住了” | 等待态逐步展示双世界、拐点、救援分支正在生成 |
| 4 | 打开结果页，先看到双世界差异摘要 | “同一个问题居然会走向这么不同的结局” | 结果页先结论后证据，差异摘要位于第一视觉层 |
| 5 | 查看关键拐点和第一句带偏的话 | “原来就是这句话、这个节点让局势失控” | 顶部第二层固定展示关键拐点并可展开证据 |
| 6 | 查看反事实救援结论 | “原来本来可以不这样，问题真的在机制里” | 顶部第三层展示救援结论，强化机制解释力 |
| 7 | 下拉查看完整 10 轮时间线 | “我不是只看了结论，我能追溯整个过程” | 时间线作为证据层服务于理解，而不是抢首屏 |
| 8 | 回到首页或切换条件重跑 | “我想换一个人格组合/机制再试一次” | 产品在结果页明确鼓励重跑，而不是把体验终结在一次结果上 |

### 时间层级设计

- 5 秒内：用户必须理解“这是一个会让普通讨论分叉失控的实验”。
- 5 分钟内：用户必须能追到“第一句带偏的话”和“如果改一个条件会怎样”。
- 长期记忆层：用户离开后仍记得“平台不是在承载讨论，而是在塑造讨论”。

### 结果页收尾

结果页最后不以“阅读结束”收尾，而以“理解已经发生，下一轮实验即将开始”收尾。

- 先用一句短而有力的结论收束这次实验，例如：
  “这次不是话题天然失控，而是热度奖励把道德审判推到了台前。”
- 紧接一个明确的再实验 CTA，鼓励用户只替换一个条件重新运行，而不是把体验终结在第一次结果上。

## 需要刻意避免的失败形态

- 失败形态 1：退化成普通 AI 论坛，只有多条回复，没有演化结构。
- 失败形态 2：退化成说教式产品，只输出结论，不让用户自己感受到滑坡。
- 失败形态 3：人格差异不明显，所有 agent 只是语气不同的同一个模型。
- 失败形态 4：经典言论不够真实，用户只觉得“这是 AI 编的”。
- 失败形态 5：阶段切换没有因果，像作者硬分段。

## 设计补充：视觉母语与 Anti-Slop 约束

首版视觉母语固定为：
`证据板 / 双世界实验台 / 互联网舆论解剖台`。

这意味着用户第一眼应该感觉自己正在查看一场被拆解、被对照、被取证的舆论事件，而不是走进一个柔和、圆角、均匀卡片的通用 AI SaaS 面板。

### 首页首屏构图约束

1. 首屏必须是一个单一构图，不是“hero + 参数区 + 功能卡片”三段拼接。
2. 主视觉锚点必须是“今日样本问题”及其双世界分叉预告，而不是品牌口号或功能卖点。
3. 唯一主 CTA 应直接指向“开始实验/开始观察”，而不是“了解更多”“查看功能”。
4. 话题改写、人格组合、热度奖励开关属于次级控制层，放在首屏下方、折叠区或抽屉中，不得与主视觉抢第一层级。
5. 首屏右侧或下缘应出现证据感元素，例如阶段条、分叉提示、第一句带偏的话预告、取证标签，而不是装饰性图标。

### 结果页首屏构图约束

1. 结果页顶部必须先像“调查结论板”，不是先露出长聊天记录。
2. 首屏固定顺序保持为：
   `双世界差异摘要 -> 关键拐点 -> 反事实救援`。
3. 完整 `10` 轮时间线下沉为证据层，服务于结论解释，而不是抢占第一视觉层级。
4. 双世界对照要通过版式差异、标签和因果说明成立，不能只靠两列卡片并排。

### 视觉词汇

- 版式以左对齐、非对称、编辑部/档案感为默认，不做全站大面积居中。
- 分隔以细线、编号、脚注、证据标签、时间戳、批注箭头为主，不靠大阴影和大圆角制造层次。
- 卡片只有在“这就是一份证据/一段摘录/一个结果对象”时才出现；禁止把整个页面切成均匀卡片墙。
- 圆角只用于按钮、输入控件和局部可点元素，控制在小圆角范围；不允许通篇统一大圆角。
- 字体需要体现“讨论平台的熟悉感 + 调查报告的判断力”：
  大标题可用 `Noto Serif SC` / `Source Han Serif SC` 这一类衬线中文，
  UI 和正文可配 `IBM Plex Sans` / `Source Han Sans` 这一类理性无衬线；
  禁止整站退化成 `Inter`/系统字体的通用 AI 产品气质。

### 颜色与动效

- 颜色基础盘应偏纸张白、墨黑、铁灰、锈红、警示黄、冷蓝，不使用紫色/蓝紫渐变默认 AI 配色。
- 红色用于失控、偏移、热度放大；冷蓝用于反事实救援、对照世界、修复可能性；不能只靠颜色区分世界，必须配合文字标签。
- 动效只保留少量有叙事意义的动作：
  1. 首次进入时，证据标签和阶段条分层出现。
  2. 结果页加载时，双世界分叉从同一基线向两侧展开。
  3. 点击拐点时，原位展开证据，不做整页跳转或漂浮特效。
- 禁止漂浮光斑、装饰 blob、波浪分割线、无目的 hover 飘动。

### Anti-Slop 黑名单

以下任一模式出现，都视为明显跑偏：

1. 顶部一个居中 hero，下面接一排功能卡。
2. 中段三列 feature grid，配“图标圆底 + 标题 + 两行说明”。
3. 页面主要层级靠统一阴影卡片堆出来。
4. 大量彩色圆形图标、装饰徽章、功能卖点贴片。
5. 默认紫色、靛蓝或蓝紫渐变的 AI 产品配色。
6. 每个 section 一样高、一样居中、一样语气，像模板拼接。
7. 结果页做成仪表盘或数据看板，而不是“结论 + 证据”的调查结构。
8. 首页第一屏把所有控制项一次性摊开，像参数玩具。
9. 世界 A/B 的差异只靠颜色，不靠文案标签、结构和因果说明。
10. 去掉阴影之后，页面就失去层次和气质。

### 自检标准

- 截图给陌生人看，对方应在 `5` 秒内说出“这是在拆一场互联网舆论怎么坏掉”，而不是“这是某个 AI 分析平台”。
- 即使删掉 `30%` 装饰，页面也应依靠排版、层级、字体和文案成立。
- 如果首页能被轻易替换成“欢迎来到 XXX AI”而不显违和，说明视觉母语还不够具体。

## 设计补充：响应式与可访问性

### 响应式

- 桌面端首页保持“主舞台 + 次级证据轨”的非对称布局；参数控制区不进入首屏主构图。
- 平板端允许把次级证据轨下移，但仍须保持“问题样本 -> 分叉预告 -> 主 CTA”这一叙事顺序，不得退化成简单卡片堆叠。
- 手机端首页保留单一主叙事流：样本问题、分叉预告、阶段条、主 CTA 依次全宽展开；控制项进入抽屉或折叠面板，避免首屏失焦。
- 桌面端结果页可以同时展示双世界摘要，但手机端详细证据层改为“固定顶层对照摘要 + 单世界详情切换”，避免把两列时间线硬压到窄屏里。
- 手机端结果页的关键拐点和反事实救援必须仍在首屏可见范围内，不能被时间线淹没到下方。

### 可访问性

- 所有主要交互控件的触控区域不低于 `44px`。
- 键盘导航顺序遵循叙事顺序：问题样本 -> 主 CTA -> 参数控制 -> 结果摘要 -> 拐点 -> 救援 -> 时间线。
- 焦点态必须清晰可见，不能只依赖浏览器默认细虚线。
- 双世界差异不可只用颜色表达，必须始终配套文字标签、标题和状态说明。
- 生成中的叙事式等待状态要有可读文本，并支持屏幕阅读器温和播报进度变化。
- 时间线应使用可被辅助技术理解的列表/分段语义，至少能读出“世界 / 轮次 / 阶段 / 发言摘要”。
- 文本对比度达到基础可读标准；如果使用淡纸色背景，正文和说明文字要单独检查对比度。
- 动效需尊重减少动态效果设置；在 `prefers-reduced-motion` 下保留层级变化，但取消分叉扩展和证据展开动画。

## 设计补充：已决与延后决策

| 决策 | 本轮结论 | 原因 |
|------|----------|------|
| 首页首屏是否先暴露完整控制台 | 否，先演示后控制 | 首屏必须先把产品读成“舆论显影实验”，否则会被误读成参数玩具 |
| 平台反馈层在 UI 中如何呈现 | 采用“叙事标签 + 因果说明”的混合方式，不先上复杂数值图表 | 首版目标是让普通用户看懂机制，不是让人阅读监控面板 |
| 手机端结果页是否双列并排展示双世界 | 否，改为顶层对照摘要 + 单世界详情切换 | 双列压缩会破坏可读性，也削弱“先结论后证据” |
| 时间线是否做成卡片瀑布流 | 否，维持连续证据轨 | 时间线是证据，不是营销模块 |
| repo 级设计系统是否已完成 | 否，延后为实现前沉淀项 | 当前缺少 `DESIGN.md`，需在实现前抽取为仓库级规范 |

## 设计补充：NOT in scope

- 首版不做营销型落地页，不需要“品牌故事 / 用户评价 / 定价 / FAQ”式外壳。
- 首版不做用户直接下场插话改变时间线的交互，优先锁定“观察、对照、重跑”主链路。
- 首版不做复杂数据图表或实时舆情监控看板，避免把产品做成分析后台。
- 首版不复刻真实平台品牌或沉浸式无限滚动信息流，防止主目标从“显影机制”漂回“刷内容”。

## Success Criteria

- 普通网友在 `30` 秒内能理解这不是普通论坛，而是一个“舆论如何变坏”的演化观察器。
- 同一个问题在不同人格组合下，确实会导向明显不同的演化轨迹。
- 用户能清楚记住 `3` 个阶段，而不是只看见一串对话。
- 每次运行至少能产出 3-5 句让用户产生强烈既视感的“经典言论”。
- 结束页能明确说出：哪里开始坏、是谁推动、平台奖励了什么。
- 用户看完一轮之后，会有动力再换一组人格重新跑一次。
